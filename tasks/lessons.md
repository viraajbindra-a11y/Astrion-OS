# Astrion OS — Lessons Learned

## Architecture
1. **Never use RGBA transparency without a compositor.** All `rgba()` in CSS and `gdk_screen_get_rgba_visual()` in C render as WHITE on X11 without compositing. Use solid colors only.
2. **`gtk_window_fullscreen()` needs a window manager.** Without a WM, use `gtk_window_resize(w, screen_width, screen_height)` + `gtk_window_move(w, 0, 0)` instead.
3. **`gtk_window_begin_move_drag()` needs a WM.** Without a WM, implement manual drag via motion-notify-event.
4. **GDK_SCALE=2 + GDK_DPI_SCALE compound.** Don't use both — GDK_SCALE alone handles HiDPI for native shell. Web content zoom is separate (webkit_web_view_set_zoom_level).
5. **ESM imports must be at the top of the file.** `import { X } from 'y'` inside a function or try/catch block crashes Node.js in ESM mode.
6. **C forward declarations required.** If function A calls function B and B is defined after A, add `static void B(...)` declaration at the top.
7. **C++ lambda syntax (`+[](...){}`) doesn't work in C.** Use named callback functions.
8. **A web page is NOT an operating system.** Users notice Ctrl+R, right-click menus, scroll bounce, and other browser behaviors. A native shell with WebKitGTK app windows is the right architecture.

## Live ISO
9. **Live ISOs lose ALL state on reboot.** localStorage, Wi-Fi profiles, settings — everything is in RAM overlay. Install to disk for persistence.
10. **System clock is wrong on live boot.** NTP needs internet, internet needs Wi-Fi, Wi-Fi needs manual setup. Use `ntpdate -u` (UDP, no SSL) in .xinitrc.
11. **DNS must be set explicitly.** Live ISOs often have empty `/etc/resolv.conf`. Write `nameserver 8.8.8.8` before anything network-dependent.
12. **SSL cert errors = wrong clock.** If `curl` fails with cert error (code 60), fix the date first: `sudo date -s "2026-04-07 12:00:00"`.

## Surface Pro 6
13. **Marvell Wi-Fi needs `firmware-misc-nonfree`.** The AVASTAR chip doesn't work without proprietary firmware blobs.
14. **Surface Pro 6 is 2736x1824 at 267 DPI.** Without scaling, everything is microscopic. GDK_SCALE=2 for native GTK, webkit_web_view_set_zoom_level(1.75) for web content.
15. **The `-nocursor` flag on `startx` permanently hides the cursor.** Don't use it. Use `xsetroot -cursor_name left_ptr` instead.
16. **`prompt()` may be blocked in WebKitGTK.** Use custom HTML dialogs instead of JavaScript `prompt()`, `confirm()`, `alert()`.

## Development
17. **`sudo nova-update` only updates web files (JS/CSS/HTML).** Compiled C binaries need a new ISO build or manual recompile on the device.
18. **Test on real hardware, not just in the browser.** Many features (battery, Wi-Fi, Bluetooth, audio) only work on the ISO.
19. **Don't hallucinate features.** Always verify with `grep` or `ls` that code actually exists before claiming it works.
20. **Cancel and retrigger builds rather than waiting for stale ones.** If you push a fix after triggering a build, the running build doesn't have the fix.

## Releases & ISO Publishing
21. **The `build-iso.yml` workflow only uploads to Actions artifacts, not to Releases.** Artifacts expire in 30 days and require login to download. For public distribution, the ISO must be manually uploaded to the Release via `gh release upload`. Fix pending: patch the workflow to auto-publish to release on tag push.
22. **Before publishing a stale ISO artifact, verify no `distro/` commits since the build.** Use `git log --oneline <artifact-sha>..HEAD -- distro/` — if it's empty, the ISO's shell/C code is still current. If non-empty, trigger a fresh build first.
23. **Always ship a SHA256 alongside any ISO.** Users need to verify integrity. `shasum -a 256 file.iso | tee file.iso.sha256` then upload both.
24. **GitHub release assets are hard-capped at 2 GB per file.** Check ISO size before attempting upload: `ls -lh file.iso`. Astrion ISO is 1.86 GB — safe, but close to the cap.
25. **Use `gh release upload --clobber` for new assets, not for replacing unrelated existing ones.** `--clobber` only overwrites assets with the same filename, leaves others alone.
26. **Name release assets with the product name + version + arch.** `astrion-os-0.1.95-amd64.iso`, not `nova-os.iso`. Makes the filename self-documenting and matches the current brand.

## Plan / Workflow
27. **The built-in `EnterPlanMode`/`ExitPlanMode` exists and is tighter than free-form plans in `tasks/todo.md`.** Plan mode enforces read-only exploration and a formal approval dialog before any edits. Use it for non-trivial tasks alongside `tasks/todo.md` (the todo file is the durable artifact, plan mode is the gate).
28. **Inversion thinking belongs in the plan itself, not just in the code.** For every task, list what will break, then invert each failure mode into the solution. This is more useful than a flat task list because it catches hidden assumptions before implementation.
29. **Commit large-file state to disk early, not at the end of a task.** Long sessions with many file edits can run out of context. Writing PLAN.md, presentation.md, etc. as soon as they're drafted (not buffered in chat) prevents losing work when context fills.

## Browser Preview & CSS Debugging
30. **`getComputedStyle(el).backgroundColor` can return stale mid-transition values.** When an element has `transition: background 0.3s` and you set its state via `dataset.brain = 's2'`, `getComputedStyle` read immediately after will return the interpolated (still in-progress) value, NOT the final target. This can make it look like CSS cascade isn't working when it actually is. Fix: either wait longer than the transition duration (e.g., 500 ms) before reading, OR use `el.computedStyleMap().get('background-color').toString()` AFTER the transition, OR use visual screenshot verification (most reliable). Inline style `!important` tests can be similarly confused by transition races.
31. **Leftover injected `<style>` tags and inline `element.style` from debugging pollute subsequent tests.** When running `preview_eval` multiple times, anything you inject persists across evals. Always remove injected styles and clear `element.removeAttribute('style')` before re-testing, OR reload the page between tests.
32. **The web Astrion OS boots through a login screen that only resolves on a click.** `js/boot.js` around line 278 has `loginScreen.addEventListener('click', ...)` that resolves the login promise when no password is set. In an automated preview environment (headless browser), the login screen never gets clicked, so `initMenubar()` and every downstream init never runs — the menubar DOM is there but has no handlers. When verifying behavior in preview: `document.getElementById('login-screen').click()` to advance past this gate. Affects every shell module that `boot.js` initializes after Phase 4.
33. **Astrion has a 5-minute idle screensaver.** If you step away from the preview mid-debug, the screensaver covers the desktop and you'll see stars + clock instead of your changes. Dismiss via `document.getElementById('screensaver').style.display = 'none'` or by triggering real input events (mousemove/keydown on the document).
34. **Files created by `sudo` in one CI step are root-owned and unusable by subsequent non-sudo steps.** `distro/build.sh` runs with `sudo`, which leaves `distro/output/nova-os.iso` owned by root. The next step in the workflow (`Publish ISO to latest GitHub release`) runs as the `runner` user and fails on `mv`/`rm` with `Permission denied`. Fix: add `sudo chown -R "$(whoami):$(whoami)" distro/output` at the top of any non-sudo step that touches files created by a sudo step. Cost is ~1 second and it's idempotent.
35. **`package.json` version drifts from git release tags in this repo.** The Electron auto-build pipeline (`build-desktop.yml`) bumps `package.json` independently, so it can say `1.0.0` while the latest release is `v0.1.95`. Always read version info from the latest release tag (`gh release list --limit 1 --json tagName`) instead of `package.json` when naming assets that must match the release. Strip the leading `v` with `VERSION="${LATEST_TAG#v}"`.
36. **"Fully opaque" comments can lie.** The radial glow in `on_desktop_draw` (nova-shell.c line ~569) had a comment "fully opaque, no compositor needed" but used `cairo_pattern_add_color_stop_rgba` with an alpha-1.0 center and alpha-0.0 edge. That IS an alpha gradient and still renders WHITE on X11 without a compositor (lesson #1). Don't trust comments — grep for `rgba`/`alpha` directly. Fix: use `cairo_pattern_add_color_stop_rgb` and pick the outer color to match the base gradient so there's no visible seam.
37. **Hardcoding `GDK_SCALE=2` in .xinitrc breaks VMs and normal displays.** On Surface Pro 6 (2736×1824) you need 2× scaling; in UTM/QEMU at 1920×1200 it makes everything 3.5× too big (combined with WebKit 1.75× zoom) and the setup wizard labels overlap horrifically. Fix: detect screen width from `xrandr`, apply `GDK_SCALE=2` only when width ≥ 2000px. User can override with `ASTRION_GDK_SCALE=1|2` env var.
38. **Fixed-column CSS grids (`repeat(3, 1fr)`) break at extreme zoom levels.** The setup wizard wallpaper picker used `grid-template-columns: repeat(3, 1fr)` inside a 560px max-width container. When the effective zoom was 3.5× in a VM, the grid cells overflowed and labels ("Geometry", "Forest", etc.) stacked on top of each other. Fix: `repeat(auto-fill, minmax(140px, 1fr))` so cells reflow, plus `min-width:0` on cells and `white-space:nowrap; overflow:hidden; text-overflow:ellipsis` on labels to truncate gracefully instead of overflowing.
39. **Read real-hardware values the same way across different subsystems.** `update_battery` reads `/sys/class/power_supply/BAT*/capacity` directly. `update_wifi` uses `popen("nmcli ...")`. `update_volume` uses `popen("pactl get-sink-volume ...")`. The pattern is identical: open, `fgets`, parse, `gtk_label_set_markup`. Reusing the pattern means no new libraries, no new dependencies, and any failure gracefully degrades to an "N/A" label. Good enough for M0.P1; libpulse integration can land in M0.P2 if we want real sliders.
40. **Verify multi-part Edit sequences after EACH step, not just at the end.** During M0.P1 Surgery 3, an Edit tool call used `new_str` instead of `new_string` (typo) and silently failed, so only HALF of the surgery landed. The call-site in the panel got added but the function definition did not, which the linker caught ~20 minutes later when CI built the ISO. Cost: one failed CI run (~27 min). Fix: after every Edit in a multi-step operation, grep for both the new symbol AND its expected use (`grep -n "update_volume" nova-shell.c` would've shown only the call sites with no definition).
41. **GtkLabel doesn't receive button-press events on its own — wrap in GtkEventBox.** M0.P2 Wi-Fi/Bluetooth/volume pickers all needed click handlers on menubar labels. Pattern: `GtkWidget *label = gtk_label_new(...); GtkWidget *evbox = gtk_event_box_new(); gtk_container_add(GTK_CONTAINER(evbox), label); gtk_widget_set_events(evbox, GDK_BUTTON_PRESS_MASK); g_signal_connect(evbox, "button-press-event", ...)`. Always pack the eventbox (not the label) into the parent container.
42. **`gtk_dialog_run()` is blocking modal — use it for simple yes/no flows like password prompts.** For the Wi-Fi password dialog, `gtk_dialog_run()` blocks until user clicks a button and returns the response code. Way simpler than connecting `response` signals. Only use for short interactions; never for long-running dialogs.
43. **Forward-declare new static functions at the top of nova-shell.c to keep the file navigable.** Put new `static` function prototypes in the "forward declarations" block near line 220. This lets you call them in any order and also documents all exported symbols in one place.
44. **Cancel superseded CI builds to save runner minutes.** When pushing multiple commits in a row, each push triggers a build but only the LATEST build matters. Cancel the in-flight older ones with `gh run cancel <run_id>` to save ~25 minutes per cancel. Rule of thumb: if you pushed a commit within the last 5 minutes of a running build, cancel the running one.
45. **Register ALL apps in native-mode boot, not a subset.** `js/boot.js`'s native branch was registering 16 of 52 apps. Result: launching any of the other 36 from nova-shell failed silently with "App not found". Always mirror the full app list in both the native and non-native boot paths, or extract them into a shared `registerAllApps()` helper.
46. **`pactl` output format varies slightly between versions.** The volume parser walks BACKWARD from the `%` character to find the integer, which tolerates different spacing. Never hard-code column positions — use anchor characters like `%` or `:` and walk relative to them.
47. **Run capability side effects BEFORE the notification, and wrap notifications in try/catch.** In M1.P2, several capability providers called `notifications.show()` at the end. When tested end-to-end in a context where the notification host DOM node didn't exist yet, the notification threw and the whole capability reported ok=false — even though the data side-effect had already succeeded. Fix: a `safeNotify()` helper that wraps the call in a try/catch. Principle: UI feedback is nice-to-have; never let it fail the underlying action.
48. **Always verify the exact storage shape the target app reads.** The notes.create capability initially wrote `{ id, title, body }` but Notes.js reads `note.content.split('\n')` — it expects a `content` field with the title as the first line. This only showed up in E2E testing. Fix: read the target app's source BEFORE writing to its storage. Lesson: "the shape you invent" almost never matches "the shape the consumer already uses."
49. **Structure capability providers so the data write happens FIRST, then the UI update.** If a notification fails or the app fails to launch, the data is still persisted and can be recovered on next open. Never couple "did the work happen" to "did the toast show up."
50. **Intent parser: handle verbs with optional targets explicitly.** "navigate to github.com", "explain recursion", "compute 5+3" all have no grammatical target — the parser needs a VERB_WITHOUT_TARGET set so these don't fail parsing. Learned during M1.P1 testing when `navigate to github.com` returned null because "github.com" isn't in TARGET_SYNONYMS.
51. **For free-form verbs (explain, summarize, ask), treat the entire remainder of the query as the topic.** Users write "explain recursion" expecting "recursion" to be the topic, not a target. In the arg extractor, set `args.topic = argsText` for these verbs before running the preposition patterns. This one line makes the parser feel natural instead of strict.
52. **Sprint compression is possible when the foundation is already solid.** M1 was scheduled for ~3 weeks per PLAN.md. It shipped in one day because M0 had already built the substrate (process-manager, file-system, event-bus, existing apps to use as backends). The lesson: invest heavily in the foundation — the rewards compound in every later milestone. M0 took ~10 days of intense work; M1 took ~4 hours because M0 had done the hard part.
53. **`crypto.subtle.digest` is async — NEVER `await` inside an open IndexedDB transaction.** IDB transactions auto-commit on the next macrotask. If you `const tx = db.transaction(...)` and then `await crypto.subtle.digest(...)` inside the callback, the microtask yield lets the txn commit (empty) before your hash resolves, and the next `tx.objectStore(...)` throws `TransactionInactiveError`. Fix: compute ALL async values (hashes, structuredClones of large blobs, network data) BEFORE opening the transaction. Inside the txn callback, use ONLY synchronous IDB calls. Learned while building graph-store.js for M2.P1.
54. **Emit events from `tx.oncomplete`, never from `request.onsuccess`.** An earlier request in a txn can succeed while a later request aborts the whole txn. If you emit `graph:node:created` from `onsuccess`, subscribers who call `getNode` inside the handler get back nothing (the abort rolled everything back). Only `oncomplete` fires after the txn is durable. Test this invariant directly: subscribe to the event, call `getNode` inside the handler, assert non-null. Caught during sanity-test design for graph-store.js.
55. **Fail-fast validation must live at NORMALIZE time, not EVALUATE time.** graph-query.js initially only caught unknown operator typos (`{ foo: { lessThan: 5 } }`) when `matchFilter` actually ran on a node. If the `from` type had zero nodes, the filter never ran and the typo was silently accepted. Fix: walk `where` at normalize time and reject any plain-object rhs with unknown keys, non-array `in`, non-boolean `exists`, etc. Rule of thumb: if an error can be detected from the QUERY alone without touching the data, detect it there. This also gives callers deterministic error behavior regardless of data state. Caught via an M2.P2 smoke test on an empty type.
56. **Prototype pollution defense requires BOTH normalize AND walker layers.** When walking a dotted field path like `'props.title'`, using `seg in cur` instead of `Object.prototype.hasOwnProperty.call(cur, seg)` means `'props.toString'` matches `Function.prototype.toString` and every node "has" it. ALSO: an attacker can pass `where: { '__proto__.polluted': 1 }` which needs to be rejected at query-parse time, not relied on at walk time. Both checks are cheap; both are necessary. Single-layer defenses leak. Caught via M2.P2 Plan agent review.
57. **One-shot migrations need a flag AND a graph-empty safety check.** graph-migration.js uses `localStorage.getItem('astrion-graph-migrated-v1')` as the primary flag, but also checks that the graph has zero nodes of the migrated types before running. Belt-and-suspenders: if someone dev-wipes the flag but the graph has data (from a previous run or manual seed), the safety check prevents double-migration. If someone dev-wipes the graph DB but leaves the flag, the flag still prevents re-running (which is usually what you want — manual resets are rare). This dual-gate pattern means "clean" resets need `window.__astrionMigrationReset()` to clear both.
58. **Graph-backed apps must subscribe to `graph:*` events for external-write consistency.** When Notes is rewired to read from the graph, and the Intent Kernel creates a note via `notes.create` capability, the capability writes to the graph but Notes's local cache is stale. Fix: subscribe to `graph:node:created` and refetch (cheap) when the type matches. Without this, a user who says "make a note called shopping" via Spotlight while Notes is open sees nothing change until they refresh. Don't couple the app to its own storage layer — couple it to the global mutation events.
59. **IDB transaction stability vs. JS engine timer drift → don't trust sub-millisecond createdAt ordering on rapid-fire creates.** Migration created 3 todos in a tight loop. Each `createNode` ran its own IDB transaction (~1-3ms). But the `Date.now()` values captured inside each createNode sometimes collided (all three landed in the same ms), which meant `orderBy: createdAt asc` returned them in arbitrary (IDB key-path) order instead of insertion order. For sub-millisecond ordering guarantees you need either an auto-increment sequence number or an explicit sleep between writes. For M2 I accepted the ordering jitter — the apps use `createdAt` ordering which is "close enough" for user-facing timelines.
60. **The rewind semantics trap: rewinding an update creates a NEW version, not a version rollback.** graph-store's `rewindMutation` on an `update_node` calls `updateNode(id, before.props)` which bumps the version counter. The node's `props` match the old state but `version` is higher than the original (e.g., v1 → v2 via user edit, v3 via rewind). This preserves the immutable provenance chain at the cost of the version counter being "append-only". Documented in the rewind method's doc comment. Callers who want EXACT version rollback need to wait for true snapshot restore in M5+.
61. **Day-by-day verification is cheaper than end-of-sprint verification.** M2 Days 3→6 were compressed into one morning. At every day boundary (migration, Notes wire, Todo wire, Reminders wire, capability-providers rewire, rewind), I ran `preview_eval` smoke tests BEFORE moving on. Day 4a caught a timing issue (700ms wasn't enough past the 500ms debounce); Day 4c caught that capability-providers.js still wrote to localStorage (bypassing the graph); Day 5 caught that rewinding deleteNode needed a low-level `_restoreNode` helper for same-id restoration. Each of those would have been a nightmare to debug if discovered at the end after 3 hours of unverified changes. The rule: every layer of new coupling gets its own verification before the next layer lands.
62. **Math detection must pre-empt verb lookup.** The intent parser's "what is 42 * 17" bug sat in the console for multiple sessions because the math-detection regex required the whole query to be a pure math expression, so "what" matched the `explain` verb synonym before math was considered. Fix: run a math pre-check BEFORE the verb scan. Strip common leading phrases ("what is", "what's", "calculate", "calc", "compute", "solve", "how much is", "tell me") + trailing `?`/`.`, then check if the remainder is a pure expression. This is a general rule: when one parser stage can shadow another, make the MORE SPECIFIC stage run first. Don't trust fallback ordering.
63. **Features you "think" are missing are often already built.** Day 1 exploration for the Polish Sprint found that the menubar clock with mini-calendar, battery percentage display, window minimize/maximize animations, focus-visible indicators, dock keyboard nav, notification history panel, Finder drag-drop, Finder inline rename, and Finder text/image previews were ALL already implemented. My original triage said "medium-large" for most of these. The ACTUAL Polish Sprint work was ~30% of the advertised scope — the rest was verification, polish, or renaming nova→astrion. The lesson: before planning a sprint, spend 20 minutes exploring the code for "already there" instead of 20 hours rebuilding. Exploration budget pays for itself 5x.
64. **An unfixed regex bug can survive 50+ code reviews.** The `"what is 42 * 17"` failure in `intent-parser` showed up in console logs for AT LEAST 4 previous commits across multiple sessions (check the git blame on the test case). Nobody — including me — fixed it until it became Day 1 of the Polish Sprint. When test failure output becomes "background noise" (1/19 always red), you stop seeing it. Fix: the regression-check at the top of every new commit should verify EVERY test suite shows "all N pass", not just "the one I touched." If a test has been red for multiple commits, FIX IT or DELETE IT — don't let it live in the background.
65. **`update-electron-app` is a complete auto-update solution; don't rebuild it.** Day 6-7 started as "integrate electron-updater from scratch." Within 5 minutes of exploration, I found `update-electron-app` was already in `installer/package.json` as a dependency AND wired in `installer/main.js:setupAutoUpdater()`. The "missing" work turned out to be: forward Squirrel events to the renderer via IPC, add a manual check-for-updates path, wire notification center, write the docs, and rebrand. None of that required touching the updater itself. Lesson: Electron's ecosystem has canned solutions for auto-update, code-signing, packaging, crash reporting, and deep-linking. Always grep `node_modules/` and `package.json` before writing anything from scratch.
66. **Accessibility infrastructure is cheap; accessibility verification is expensive.** Day 8 added 13+ aria-labels, role attributes, tabindex, a focus-trap utility, and retrofitted one dialog. All of that took ~30 minutes. But REAL accessibility — testing with VoiceOver on macOS, Orca on Linux, NVDA on Windows, real keyboard-only navigation of every screen — is a multi-session effort that requires being able to HEAR the screen reader output. For v0.2 the right move is: ship the infrastructure, don't claim "screen reader compatible" until it's been tested. False accessibility claims are worse than no claims — they mislead users who genuinely depend on assistive tech.
67. **Focus traps need capture-phase listeners.** A naive focus trap uses `document.addEventListener('keydown', handler)` (bubble phase). Dialog children can call `stopPropagation()` in their own keydown handlers (e.g., an input canceling Enter/Escape) and the trap never runs. Fix: use `document.addEventListener('keydown', handler, true)` — capture phase runs BEFORE any descendant listener. The tradeoff is that you have to be careful about `e.preventDefault` so you don't break inputs.
68. **`requestAnimationFrame` before calling `.focus()` on a newly-mounted element.** If you call `el.focus()` synchronously after `document.body.appendChild(dialog)`, the browser hasn't painted the dialog yet; scroll-into-view can misbehave and screen readers may not pick up the focus change. Wrapping `focus()` in a `requestAnimationFrame` callback lets the browser paint once before focusing. Tiny detail, big UX difference on slower devices.
69. **Window Management API requires explicit user permission — never pre-fetch on boot.** `window.getScreenDetails()` (Chromium 100+, needed for multi-monitor positioning) triggers a permission prompt the first time it's called in a browsing session. If you call it in `windowManager.init()`, every user sees a "Allow this site to manage windows on all your displays" prompt on first load — a terrible first impression. Correct pattern: expose a lazy `refreshScreenDetails()` method and only call it when an app explicitly needs multi-display info (Mission Control, Display Settings, etc.). The fallback single-display path uses `window.innerWidth/Height`, which is correct for single-monitor setups anyway.
70. **Sprint compression works when a prior sprint already did the foundation work.** Lesson #52 applied to M1. Lesson #63 is its flip side. The Polish Sprint that was budgeted for 9 days shipped in 1 day because (a) M0/M1/M2 had already built so much of the underlying infrastructure, and (b) Viraaj's checklist conflated "this exists" with "this needs to be built." Every sprint should start with an exploration pass that catalogs what's already there. If the actual delta is less than 30% of the advertised scope, budget accordingly — don't pad the timeline just because the list looks long.

## Agent Core Sprint (v0.3)
71. **ESM module identity is keyed by the FULL RESOLVED URL, not by file path.** When `preview_eval` called `import('/js/kernel/event-bus.js')` (absolute) and boot.js had `import './kernel/event-bus.js'` (relative), both should have resolved to the same URL and therefore been the same module instance. But adding a query string like `'/js/kernel/intent-executor.js?v=2'` for cache busting DOES create a new URL → a new module graph → a completely separate instance (different eventBus, different aiService singleton, different capability registry). This caused hours of phantom "why isn't my listener firing" debugging during Agent Core Sprint Phase 6. Rule: (a) never query-string-bust imports if you're depending on singleton identity with already-loaded code, (b) always reload the page to pick up a changed module, and (c) when debugging event-bus-crossing code, first confirm both sides are on the same module instance with `a.eventBus === b.eventBus`.
72. **`localStorage['nova-ai-provider']` can silently neuter every AI path.** During Agent Core Phase 6 verification, `aiService.ask()` was falling straight to `_mockResponse` without ever calling fetch to `/api/ai` or `/api/ai/ollama`. I spent ~45 minutes stubbing fetch, intercepting at the network layer, and re-reading the ai-service code before noticing the cause: `localStorage.getItem('nova-ai-provider') === 'mock'`, probably set by a Settings > AI test in a prior session. With `provider = 'mock'`, both `if (provider === 'auto' || provider === 'ollama')` and `if (provider === 'auto' || provider === 'anthropic')` skip, and the code falls straight to the offline mock string. Fix for verification: `localStorage.removeItem('nova-ai-provider')` before any AI-dependent test. Structural fix for later: raise a visible warning in the menubar brain indicator when provider is forced to 'mock' so users know why nothing works.
73. **Stub the singleton method, not the global.** To simulate Claude's response in a preview environment without an API key, I tried three approaches: (a) `window.fetch = stub`, (b) importing aiService and replacing `aiService.ask`, (c) replacing `aiService._mockResponse`. Only (c) worked cross-module because: (a) the stub misses when `provider='mock'` (see #72); (b) the import returned the same singleton BUT only because the URL matched exactly (lesson #71); (c) patches the singleton's own prototype method so every module sees the same override. Lesson: when patching for tests, prefer patching the concrete method on the singleton instance over swapping globals or intermediate layers. It's resilient to module-graph surprises and provider-config accidents.
74. **The intent-planner's two biggest risks are the same risk: structural validation of LLM output.** Malformed JSON and hallucinated capability ids are both "the LLM said something that isn't true about the system." The planner handles both in the same place: `tryParseJSON()` tolerates markdown fences + leading prose, then `validatePlan()` walks every step and calls `getCapability(step.cap)` to ensure it's in the registry. One retry with the error echoed back covers the "LLM was sloppy" case; the second failure returns a graceful `{ status:'failed' }` that Spotlight surfaces as a soft error. Rule: every LLM boundary in Astrion needs a strict schema check and a bounded retry — no "best effort" parsing, no fall-through-and-pray.
75. **Plan preview gating on LEVEL.REAL is the M2 precursor to the full Socratic loop (M6).** The executor's `executePlan()` sums total irreversibility tokens up-front and, if any step is L2+, emits `plan:preview` and waits for `plan:confirmed` OR `plan:aborted` before running a single step. Spotlight renders the preview with yellow border + "⚠ Press ↵ Enter to confirm" header. This is cheap to implement (~40 lines in the executor + ~15 lines of Spotlight render code), costs nothing for pure-sandbox plans (folder/file creates are L1 and skip the gate), and gives the user a real choke-point on dangerous operations well before M6 ships the full red-team agent. Never defer safety primitives to "the right milestone" when the minimum viable version is this cheap.
76. **Event-driven step streaming is more ergonomic than async iterators for UI code.** Early draft: `for await (const step of executePlan(plan))`. That's clean in the executor but forces Spotlight into an async loop that's hard to abort cleanly and doesn't compose with the existing event bus pattern. Final design: `executePlan` emits `plan:step:start`, `plan:step:done`, `plan:step:fail`, `plan:completed`, `plan:failed` on the shared event bus. Spotlight subscribes in `initSpotlight` and re-renders the panel on every event. Benefits: (a) any code can subscribe — not just Spotlight — so the native-mode shell could hook the same events; (b) Escape-to-abort fits naturally as another emit; (c) no generator state to leak. The pattern reuses the M1 intent lifecycle events exactly. Consistency matters.
77. **Heuristic routers beat "always use the LLM" for single-shot queries.** The planner is expensive (~1-3s Claude round-trip) and the existing parser is ~2ms. Routing every query through the planner would make "open terminal" feel broken. The `routeQuery` heuristic takes a query + parser result and picks fast-path OR plan-path in <1ms: compound markers (`and`, `then`, `;`), plus a verb-hints list for " and X" clauses, plus a confidence-threshold fallback for queries the parser rejected. This preserves the M1 snappy feel for the 80% single-capability queries while unlocking multi-step for the 20% that need it. Measure the cost of each path before routing — don't use the expensive path for everything just because it's "smarter." #52 / #70 applied to latency.
78. **Session-only conversation memory is the right privacy default.** Memory is stored in the M2 hypergraph as `conversation-turn` nodes, each tagged with a `sessionId` that rolls on 10-minute idle. The planner only ever fetches turns from the CURRENT session. Rejected alternatives: (a) persistent global memory leaks context across unrelated tasks ("I was helping yesterday with budgets" bleeds into today's coding); (b) no memory at all breaks "and now do the same for Documents" follow-ups. Session-only is predictable, graceful, and still M5-compatible (rewind works on turn nodes because they're just graph mutations). Don't let memory scope creep without a deliberate design decision — memories that outlive their purpose become privacy footguns.
79. **Binding substitution belongs in the EXECUTOR, not the planner.** I briefly considered having the planner emit fully-resolved args by inlining the expected folder path ("/Desktop/Projects") into step 2's `parent`. Rejected because: (a) the planner's expected path might not match the capability's actual output (e.g. if path resolution canonicalizes differently), (b) hard-coded paths make plans non-portable across reruns. Instead, step 1 declares `binds: 'folderPath'`, step 2 references `${binds.folderPath}`, and the executor resolves the binding from step 1's ACTUAL output right before running step 2. Generalization: anywhere a plan has cross-step data flow, make the upstream step name its output and the downstream step reference that name — don't let either side assume what the other produced.
80. **Every LLM boundary needs an offline fallback path that's verifiable.** The deliverable test couldn't rely on a real Claude API call because the preview sandbox has no `ANTHROPIC_API_KEY`. Instead of shipping code that "probably works when deployed", I verified the full planner → executor → files on disk → Spotlight panel pipeline by stubbing `aiService._mockResponse` to return a canned plan. This is NOT a substitute for real Claude testing — it's a complement. The real Claude round-trip has to be verified in a later session with a key set. Rule: build testability into the LLM boundary from day one (stubbable, inspectable, deterministic under stub) so you can ship the surrounding logic without waiting on credentials.

## Agent Core Sprint — follow-up review (v0.3-review)
81. **"Defensive readers, never throws" is a CLAIM you have to adversarially test, not a design assertion.** The prior session's retrospective said `summarizeContext()` in `context-bundle.js` "returns null, never throws." A follow-up review found a one-line path where it absolutely DID throw: `new Date(bundle.timestamp).toISOString()` blows up with `RangeError: Invalid time value` when `bundle.timestamp` is `undefined`/`null`/`NaN`. The error propagated up through `buildPlannerPrompt` → `planIntent` → the `intent:plan` handler's `catch` block, which only did a `console.warn` — so the whole plan silently died with zero `plan:*` events. The prior session never hit it because every call site in the sprint passed `getContextBundle()` (which always sets a real timestamp). Any adversarial test with a bare `{}` context would have caught it in 30 seconds. Rule: for every "this never throws" / "this never returns null" claim in a retrospective, write a test that deliberately feeds it the bad input BEFORE you write the sentence. The claim IS the test spec. The catch-and-console.warn pattern in an async event handler is especially dangerous because it silently hides these crashes from everywhere except the console.
82. **Unresolved template bindings should fail fast with a clean error, not leak into downstream capability args.** `resolveBindings()` in `intent-executor.js` has a deliberate "leave unresolved for validation" fallback where `${binds.nope}` stays literal if the upstream step didn't produce a matching binding. The intent was "the capability's validate will catch the garbage path." In practice it caught it — but with the confusing error `"Path outside allowed roots: /${binds.nope}/file.txt"`, which has nothing to do with why the plan really failed (a typo'd binding name in the planner's JSON output). Fix: scan the resolved args for any surviving `${binds.X}` tokens in `executePlan` BEFORE calling the capability, and emit `plan:step:fail` with `"unresolved binding: ${binds.X}"`. Generalization: anywhere you have a substitution layer that can leave sentinels in the output, add a post-substitution guard that fails with an error naming the sentinel. Downstream code has NO context about substitution — the failure message should never come from there.
83. **Stub verification + sanity tests ≠ shipped, even when the stub suite is thorough.** (see below for full text)
84. **Local LLMs ramble past the JSON close-brace if maxTokens is too generous.** The planner's `maxTokens: 1500` was tuned for Claude Haiku which stops cleanly after the JSON object. qwen2.5:7b kept generating prose explanations after the closing `}`, pushing response time past 30 seconds and sometimes triggering the AbortSignal timeout. Fix: cap at 500 tokens (a 5-step plan is ~300 tokens). Generalization: when you tune a parameter for one model and then swap models, re-evaluate every parameter that affects output length, temperature, and format compliance.
85. **Schema-valid but semantically wrong plans are harder to catch than schema-invalid ones.** "delete everything" produced a structurally valid plan that created a folder called `_delete_me` — the model obeyed "use only catalog capabilities" but MISAPPLIED them to approximate an intent it couldn't satisfy. The fix was adding a prompt rule: "if the user's intent cannot be satisfied by any combination of catalog capabilities, return clarify instead of forcing a plan." But this is a band-aid — the real defense is an intent-alignment validator that checks whether the plan's SEMANTIC effect matches the user's stated intent, not just whether the capability IDs are in the registry. That's M6 (Socratic loop) territory.
86. **Spotlight's `input.disabled` state must be reset on EVERY exit path, not just the happy path.** `handleSubmit` disables the input when entering the plan branch. `plan:completed` re-enables it after 1.2 seconds. But clarify, abort, and failure paths never re-enabled it — the input stayed disabled on the NEXT Spotlight open, locking the user out permanently. Fix: `open()` and `close()` both set `input.disabled = false`; the `plan:clarify` handler does the same before rendering choices. Rule: for any state flag that blocks user interaction (disabled, loading, locked), list every function that exits the interaction mode and assert the flag is reset in each one.
87. **Soak-testing with a weaker model is MORE informative than soak-testing with the model you tuned for.** The planner prompt was tuned for Claude Haiku 4.5. Testing with qwen2.5:7b found bugs that Haiku would have masked: maxTokens rambling (#84), semantically wrong plans (#85), and timing-dependent UI bugs (#86) that only surface when responses take 5-15 seconds instead of 1-3. A model that follows instructions 95% of the time validates the happy path. A model that follows instructions 70% of the time validates the ERROR paths — which is where real users live. Use the weaker model for soak testing, the stronger model for production.

## Phase 0 — Chat Foundation (Agent Core Expansion)
88. **`aiService.conversationHistory` pollution breaks cross-concern callers.** The intent planner calls `aiService.ask(plannerPrompt)` which adds the planner system prompt to conversationHistory. The next `aiService.ask('What is the capital of France?')` sees the planner prompt in history and the model continues outputting JSON instead of a natural reply. Fix: add a `skipHistory` option to `aiService.ask()` — when true, don't include prior turns and don't record the exchange. Use it for all planner calls and summary-generation calls. Rule: any code path that uses `aiService.ask()` for a non-conversational purpose (planner prompts, summary generation, internal tool calls) MUST skip history to avoid poisoning the chat context.
89. **Don't use `executePlan()` from non-Spotlight contexts — it emits events Spotlight subscribes to.** `executePlan()` emits `plan:started`, which triggers Spotlight's handler to call `open()`. If Messages uses `executePlan()`, Spotlight would hijack the UI on every actionable message. Fix: Messages has its own inline mini-executor (`executeStepsInChat`) that calls `cap.execute()` directly without emitting plan events. Binding resolution is shared via exported helpers from intent-executor.js. Rule: when a module emits events as part of its API contract, any new consumer must check ALL subscribers for side effects before calling it.
90. **`routeQuery`'s VERB_HINTS list must include every verb users actually say after "and".** The compound query "...and tell me a joke" was misrouted to fast path because `tell` wasn't in VERB_HINTS. Only caught by the inline sanity test. When you add a routing heuristic based on keyword lists, seed the list from REAL user queries, not just from capability verbs.
91. **A `_processing` guard must cover the entire async lifecycle and reset in `finally`.** Without a guard, rapid sends interleave two concurrent `sendMessage` calls that corrupt shared state (conversation array, typing indicator). With a guard, the `finally` block MUST reset it — otherwise a single error permanently locks the input. The pattern: `if (_processing) return; _processing = true; try { ... } finally { _processing = false; input.disabled = false; }`.

## Full Codebase Audit (Phase 0 follow-up)
92. **User queries injected into LLM prompts must be JSON.stringify'd, not template-literal'd.** The planner prompt had `USER QUERY: "${query}"` — a crafted query containing quotes, backticks, or JSON can break the prompt structure and inject fake planner rules. Fix: `USER QUERY: ${JSON.stringify(query)}`. Rule: every LLM prompt boundary that includes user text must escape it as if it's untrusted data, because it is.
93. **Path traversal checks must run BEFORE normalization, not just after.** `resolveVfsPath` normalized paths (collapsing `/../`) before `isPathWithinRoots` checked for `..`. After normalization, `/Desktop/../etc` becomes `/etc` — the `..` is gone and the guard passes. Fix: reject `..` in the raw input BEFORE any normalization, AND check again after. Belt-and-suspenders. Also: restrict leaf names in `joinVfsPath` to reject slashes entirely.
94. **Budget checks and budget recording must be atomic (single read-modify-write).** Two concurrent `executePlan` calls can both read the same remaining budget, both pass, and both execute — spending 2x the limit. Fix: `reserveBudget()` does a single localStorage read → check → write in one synchronous call. This isn't truly atomic (no mutex), but localStorage is synchronous and single-threaded in a browser tab, so it's safe within one tab.
95. **Every event bus `emit()` must wrap each listener in try/catch.** One listener throwing an exception kills all subsequent listeners in the chain. A bug in any subscriber (even a UI module) can silently break the intent executor, graph store, or any other critical path that emits events. Wrap each callback independently.
96. **`registerCapability()` must warn on duplicate IDs.** Silent overwrite lets a malicious or buggy plugin replace `notes.create` with a trojan. At minimum: `console.warn`. At best: throw.
97. **Migration functions need a "migrating" lock in addition to the "done" flag.** Without it, two browser tabs opening simultaneously can both pass the flag check and run migration in parallel, creating duplicate data. The lock (`localStorage.setItem('migrating', 'true')`) is checked synchronously before any async work begins, and cleared in the finally block.

99. **PLAN.md milestone status drifts from disk faster than retrospectives.** PLAN.md listed M3 as "Pending — ~4 weeks." A 30-minute audit found `budget-manager.js`, `calibration-tracker.js`, ai-service.js Ollama+Anthropic+budget+escalation routing, and Settings dashboard for both budget + calibration ALL shipped. The lesson-#98 rule generalizes: don't trust ANY status column in PLAN.md — grep before billing yourself for work you may have already done. Update the plan IMMEDIATELY when you ship, not at end-of-sprint, or future you spends a session re-implementing what past you finished.
100. **The "actual brain" is in the event payload, not the provider setting.** intent-executor.js was reading `localStorage.getItem('nova-ai-provider')` to tag calibration samples — but `auto` falls through Ollama→Anthropic so the static provider check tags every Anthropic-served plan as `s1` (Ollama). The intent:plan handler is sequential within one query so a single module-level variable is race-free for the in-flight plan. Rule: any "what answered the last call?" decision must read the call's actual outcome, not the user's saved preference.
101. **`/app/:appId` should hand-pick CSS, not include the union.** Original route loaded 17 hard-coded CSS files (covered ~17 of 80 apps) and used `display:none` overrides to hide shell chrome. New route checks `existsSync('css/apps/${appId}.css')` and only includes that one file (plus system + window). Side win: blocks path traversal via a `[a-z0-9-]+` whitelist on the route param so `/app/../../../etc/passwd` returns 400. Rule: serving "everything just in case" is a load-time tax AND a maintenance debt — every new app silently breaks unless the include list is updated. 102. **Verification suites need fileSystem.init() AND graphStore.init() before any capability touches data.** The first run of test/v03-verification.html crashed at `executePlan` with `Cannot read properties of null (reading 'transaction')` because graph-store was init'd but file-system wasn't. capability-providers/files writes to fileSystem first, then graphStore. 103. **Provenance preserves pre-resolution args; tests checking "binding was resolved" must inspect the resolved args, not the result blob.** A naive test asserts `!JSON.stringify(result).includes('${binds.X}')` — but the executor stores the original placeholder under `result.provenance._intent.args` for audit, so the assertion always fails. Correct test: assert `result.provenance.args.parent === '/Desktop/Real-Path'`. Provenance is INTENTIONAL pollution; design verification around it.
104. **The Ollama installer (`curl … | sh`) can fail inside debootstrap chroot due to network/DNS quirks; tolerate it.** build.sh wraps the install in `|| echo "..."` so a missing Ollama doesn't kill the ISO build. Same pattern as Waydroid + linux-surface installs above it. The runtime degradation is graceful: ai-service.js's `_tryOllama` returns `null` on connection failure and the router falls through to Anthropic or mock. Rule: chroot package installs that fetch from external networks must always be `|| true`-guarded; never let an optional component fail the whole build.
105. **Tiny hand-rolled parsers beat `Function()` for any DSL with <10 operators.** ai-service.js's mock math handler was using `Function('"use strict"; return (' + expr + ')')()` with a regex-gated input. The regex made it safe in practice, but `Function()` is still a code smell that fails security audits. A 35-line recursive-descent parser (tokenize → expr → term → factor) handles +, -, *, /, parens, unary minus with no eval surface and similar perf. Rule: every use of `eval`/`Function()` deserves a 30-minute look at writing the parser inline — it's almost always less code than you think.
106. **Module-level event-bus state is a race-trap; per-call meta is the safe pattern.** The first attempt at fixing the brain-detection bug (lesson #100) used a module-level `__lastBrain` variable updated from `ai:response`. That was correct for sequential intent:plan handlers but breaks the moment two plans run concurrently — the second plan's brain clobbers the first's calibration sample. Real fix: add `aiService.askWithMeta(prompt) → {reply, meta}`, propagate `meta` through `planIntent`'s return value, read `plan.meta.brain` in the executor. The global event bus is fine for UI subscribers (menubar brain indicator) but never for "what was THIS specific call's outcome?" decisions. Rule: a module-global "last X" is a smell whenever multiple concurrent operations can write to it. Default to per-call return values.
107. **Verification-page hangs are usually duplicate `const` declarations across module-style top-level await blocks.** `test/v03-verification.html` was rendering `Running tests…` indefinitely with zero `.test` divs and zero h2 sections — meaning the script never executed. The error was a SyntaxError (`Identifier 'planIntent' has already been declared`) thrown at parse time, before the first `section()` call. Browser DevTools usually surface this on the console immediately, but in headless preview it's silent. Diagnostic: fetch the script body, base64-encode it into a `data:text/javascript` URL, `await import(url)`. The dynamic import will throw the parse error with a clean message. Rule: when a `<script type="module">` runs nothing visible, never trust "module loaded fine" reports — round-trip the source through a data URL import to surface the real error.
108. **Verify a handoff claim before re-implementing what it lists.** The session handoff said "Timer leaks remain in 7+ apps … no MutationObserver cleanup on window close." A 2-minute audit (`grep -nE "MutationObserver|isConnected" js/apps/{clock,music,beat-studio,animate,matrix-rain,youtube,voice-memos}.js`) showed every one of those apps already had a complete cleanup observer that cleared its timers. The handoff was wrong. Rule: a handoff bullet that names specific files to fix becomes a falsifiable claim. Run the falsifying check (read the cleanup code) BEFORE planning the fix. This is lesson #98 / #99 applied to handoff documents instead of plans.

## M4 spec→tests→sandbox sprint
109. **Spec-first / tests-from-spec / code-to-pass-tests is an order, not a guideline.** Tests generated from already-written code prove nothing — they describe the code's behavior, not the user's requirement. M4.P2's `generateTests` rejects DRAFT specs (only frozen ones can produce tests) so users can't accidentally test against criteria that haven't been approved. The freeze gate is L2 in the capability registry — users have to explicitly accept the spec. This is the real safety contract; LLM hallucinations are caught at the spec stage where they're cheap to fix, not at the code stage where they're already in the dock.
110. **Stub ordering matters when prompts share keywords.** `test/v03-verification.html` section 9 stubs `aiService.askWithMeta` to return canned spec output for prompts containing "spec generator" and canned test output for prompts containing "test generator". Both prompts end up containing some of the same words — if you check the wrong substring first, the planner's prompt cascades into the wrong branch and the test silently produces a useless spec. Always check the MORE-SPECIFIC marker first. Same pattern as the BAD-INTENT shadow case (which `spec generator` would have shadowed). Generalization: when stubs route on prompt content, order branches by specificity not by chronology.
111. **Schema-validate generated test code at the boundary, never at execution time.** `test-generator.js`'s `validateSuite` rejects `import|require|fetch|XMLHttpRequest|WebSocket|eval|Function` tokens in the generated test bodies before they ever reach the runner. Defense in depth — even though the runner uses an iframe sandbox with `allow-scripts` only (no `allow-same-origin`), an LLM-supplied test that *tries* to fetch is a sign of a buggy plan and should be flagged immediately. Two layers of defense: schema validator (fast, catches dumb mistakes) and sandbox (slow, catches everything the validator misses). Don't rely on either alone.
112. **`sandbox="allow-scripts"` without `allow-same-origin` is the right default for LLM-generated code.** The iframe gets a unique opaque origin. It can run `<script>` (so test code executes) but `localStorage`/`sessionStorage`/`indexedDB`/`document.cookie` all throw `SecurityError` and `parent`/`top` are cross-origin (no synchronous access to the host). The verification suite has an explicit "sandbox blocks parent localStorage access" test that asserts `localStorage.getItem("nova-ai-provider")` throws inside the sandbox — confirms the isolation is real, not just theoretical. This is the M4.P3 substrate that future code-gen iterations rely on.
113. **postMessage + per-call result correlation beats async iterators for sandboxed work.** `test-runner.js`'s sandbox communicates by `parent.postMessage({ type: 'result', index, ok, ... })`. The parent's `runSuite` loop awaits each test sequentially, matching results by `index`. This is uglier than `for await (const result of suite)` but it (a) survives an iframe that loads but never replies (bounded by per-test timeout), (b) reuses one sandbox for the whole batch (cheaper), (c) maps cleanly to the existing graph-store recordSuiteRun() shape. Async iterators are a worse fit when you need timeout + correlation + persistence.
114. **`recordSuiteRun` should compute pass count, not the caller.** `test-runner.runSuite` does the work then calls `recordSuiteRun(suiteId, results)`. `recordSuiteRun` itself filters `results` to count passes and stores `lastRunPasses` + `lastRunTotal` on the node. Counting twice (once in the caller, once in the storage layer) is a recipe for desync. Single source of truth: the storage helper IS the pass-count authority.

## M4.P3.b code-from-tests + M4.P4 promotion gate
115. **Sandbox bootstrap should accept a `'load'` step that parses code at load time, not at run time.** `test-runner.js`'s sandbox grew a `'load'` message that does `new Function(payload.code)` to syntax-check the shared code BEFORE the first test runs, then stores it for prepending to every test body. If the load throws, the sandbox posts back `'load-error'` and the whole suite fails up-front with one clean error instead of N "ReferenceError: App is not defined" failures. Rule: when an iframe sandbox needs shared bootstrap state, validate it at load time and surface a single clear error — never let bad code mask itself as N test failures.
116. **The code generator's failure-loop input is the failure LIST, not the failing test set.** `generateCode` builds the retry prompt from `lastResults.results.filter(r => !r.ok)` — index, title, error message per failure. The model doesn't need the original code re-pasted (it's still in context window) but it DOES need the specific error messages so it can connect each failure to the change required. Sending only the count ("2 of 3 failed") is too vague; sending the entire suite re-prints information the model already has. The shape is "here is exactly what is broken, fix it" — nothing more.
117. **Defense-in-depth tokens for code generation: the runner's sandbox + the generator's validator + the test generator's validator all enforce overlapping forbidden lists.** test-generator rejects `import|require|fetch|XMLHttpRequest|WebSocket|eval|Function`. code-generator rejects the same PLUS `setTimeout|setInterval|setImmediate|importScripts|document|window.parent|window.top`. The sandbox itself blocks every network/storage call by virtue of unique-origin. Each layer alone has gaps; the union covers prompt-injection (test-generator), buggy code-generation (code-generator), and post-generation runtime escape (sandbox). Don't collapse them into a single check — three cheap checks at three boundaries beat one expensive check at one.
118. **Test-driven code generation needs a hard cap on attempts AND a bound on per-attempt token budget.** `generateCode` defaults to `maxAttempts=3`, capped at 5. Each attempt costs ~2000 output tokens. Without the cap, an LLM that consistently misses a constraint can chew through arbitrary budget while making no progress. With the cap, worst case is 3 × 2000 = 6000 output tokens per spec — well below the M3 per-intent budget cap ($0.05) for both Haiku and Sonnet at 2026 prices. Rule: any iteration loop with an LLM in the cycle must have a hard attempts cap AND a per-attempt token cap. Compose the two — neither alone is sufficient.
119. **App promotion is the right milestone to teach Astrion that "shipping" requires explicit human assent.** `app-promoter.bundleApp` is L0 (free, no user prompt) — it's a pure graph write that documents the artifact. `app.promote` is L2 (user approval gate) — moving an app from sandbox to dock requires explicit consent. The same gate hooks where M6's red-team agent will plug in later: a promotion plan triggers the L2+ preview, the user OR red-team can reject. By making the gate an L2 capability today, the M6 retrofit is trivial — it adds another check, doesn't restructure the flow.
120. **Provenance edges make the answer to "where did this come from?" cheap.** `bundleApp` writes three edges: `app -[derives_from]-> spec`, `app -[passed_tests]-> suite`, `app -[runs_code]-> code`. Spotlight can answer "show me the spec for the app I just installed" with a single graph hop. Without edges, the same answer requires walking from app.specId → spec node, app.suiteId → suite node, etc. — same data, slower retrieval. When you persist a back-pointer in the props, also persist it as an edge; the storage cost is trivial and query plans get much simpler.

## M5.P1 branching storage substrate
121. **Transaction logs beat forked graphs for branching at ~1000+ node scale.** Forking the entire IndexedDB graph on every L2+ action would multiply storage by N branches and slow queries by O(N) lookup-across-branches. A transaction log per branch (createNode/updateNode/deleteNode/addEdge calls + their args) keeps storage flat: each branch is one extra graph node with `pendingMutations[]`. The replay-on-merge model also preserves `createdBy`/`capabilityId`/`meta` provenance verbatim because we re-call the original mutation API rather than reconstructing state. Tradeoff is conflict detection (M5.P3 base-version check) but for single-user single-tab the last-writer-wins behavior is fine.
122. **Pre-bound `record` closures in branch handles eliminate the "did I pass the right branch id?" footgun.** `createBranch()` returns `{id, status, record}` where `record` is `(mutation) => recordMutation(id, mutation)` — already bound. Callers write `await b.record({ kind: 'updateNode', args: {...} })` without repeating the branch id. Cuts a class of bugs (typo'd ids, wrong-branch writes) at the API surface, not at the validator. Apply this pattern wherever a long-lived handle owns a piece of identity that every operation on it needs.
123. **`mergeBranch` returning `{ok, applied, failedAt, error}` instead of throwing on partial failure makes "rebase and retry" possible.** A naive merge throws on first failure and the caller can't tell how far it got. Returning the count + failure index means the user can see "applied 3 of 5; step 4 failed because target node was deleted" and decide: discard the branch, edit the remaining mutations, or re-run from `failedAt`. Same pattern in `code-generator`'s history (per-attempt results array) — partial-progress reporting is more useful than all-or-nothing for any iterative operation.

## M5.P2 operation interceptor
124. **Per-call opaque ids beat global per-event correlation when multiple confirmations can be in flight.** `requestConfirmation` mints `'icpt-' + Date.now() + '-' + random(6)` per call and emits/listens for that id specifically. With a global "pending" event channel, two L2+ ops fired in quick succession would race for the next `interception:confirm` and one would resolve the wrong promise. Opaque per-call ids isolate each interception. Same pattern as the test-runner's per-test `index` correlation (lesson #113): when a sandbox/UI/kernel boundary can have multiple in-flight requests, ALWAYS attach a per-call id to both directions of the message round-trip.
125. **A hard cap on pending interceptions (`MAX_PENDING = 32`) prevents the interceptor from becoming a memory leak.** Without it, a buggy subscriber that never replies turns every L2+ call into a Promise that holds a closure forever. Hard fail at the cap with `reason: 'too many pending interceptions'` so the caller gets a fast failure instead of a hung capability. The same rule applies to ANY queue/map of in-flight async operations: bound it, don't trust subscribers to drain.
126. **Generalising a feature from "one caller" to "any caller" is usually a one-page rewrite, but only AFTER the one-caller version has run for a while.** The M2 Agent Core L2+ preview gate lived inside `intent-executor.executePlan` for several months as a private helper. M5.P2 lifted the same logic into `operation-interceptor.js` with two changes: per-call ids (instead of relying on a single in-flight plan), and a high-level `interceptedExecute(cap, args)` wrapper. The shape was already proven; the generalisation took ~120 lines. Don't generalise speculatively — wait until the one-caller version is solid, then lift.
127. **`opts.skipInterception` is a footgun that's worth shipping anyway.** Bypassing the user-confirmation gate defeats M5's whole purpose. But there are real headless/automation contexts (background agents, scheduled tasks, the v0.3 verification suite itself) where the gate hangs forever. ## M5.P2.c Spotlight UI subscriber
128. **Enter on an empty input is a real input event when a confirm gate is pending.** Spotlight's input keydown handler used to early-out on `if (query) handleSubmit(query)`. With the M5.P2.c interception flow, the user presses Enter without typing anything because the panel itself IS the input. The fix: also call handleSubmit when a confirm gate is pending (`pendingInterceptionId || pendingConfirmPlanId`), even on empty query. Generalisation: input handlers that "have nothing to do unless the user typed something" assume keystroke = user-text-input, which is wrong whenever the keystroke is also an affordance for a different ambient state.
129. **UI subscribers should react to events even when no other state matches.** The interception preview subscriber doesn't check `if (this is my preview)` — there is no scope. ANY interception:preview event opens Spotlight and renders the panel. This is correct: the operation-interceptor just emitted a request for user attention, and the UI's job is to give it. If multiple previews come in flight, the cap of `MAX_PENDING = 32` in operation-interceptor.js bounds the queue; the UI's last-write-wins behavior is fine because the user can only act on one at a time anyway.
130. **Functional event-flow verification > visual screenshot for headless gates.** The M5.P2.c verification couldn't easily screenshot the live Spotlight panel (the page also boots the setup wizard which obscures it on fresh load). Instead the test dispatched `KeyboardEvent('keydown', { key: 'Enter' })` and `'Escape'` programmatically and asserted on captured event payloads (`captured: [{type: 'confirm', id: 'icpt-c1'}, ...]`). For UI features whose contract is "emit X event when user does Y", testing the event payload is more reliable than testing pixels. ## M5.P3 rewindBranch
131. **`graph-store.updateNode` reads `meta.capabilityId` directly; `createNode` reads it from `_sanitizeMeta(meta)` which drops nested objects.** The two functions look identical at the call site but treat their `meta` parameter differently. updateNode reads `_sanitizeMeta(meta).capabilityId`; createNode reads `_sanitizeMeta(meta).capabilityId` after _sanitizeMeta turns the whole meta into createdBy. Net effect: pass `meta = { capabilityId: 'X' }` (top-level) and BOTH paths record it. Pass `meta = { createdBy: { capabilityId: 'X' } }` and only createNode picks it up — updateNode silently drops it. The first M5.P3 attempt nested capabilityId under createdBy (intuition from how the node's createdBy field looks at rest), and rewindBranch found zero tagged mutations because updateNode never wrote the tag. Generalisation: ALWAYS check the destination function's parameter handling before assuming a nested shape will round-trip.
132. **Tagging mutations with `capabilityId = 'branch.merge:' + branchId` is enough; you do NOT need to capture mutation IDs at merge time.** The first design intuition was "capture every produced mutation id during merge, store them on the branch node, then rewind by id list." Simpler: tag the meta with the branch-specific capabilityId, then at rewind time query graph-store for mutations with matching capabilityId. The mutation log is already an index by capabilityId for free. Saves a write per mutation + simplifies the branch node schema. Lesson: when a downstream system ALREADY indexes by some field, propagate that field rather than inventing your own bookkeeping.
133. **Status='rewound' as an idempotency guard is more useful than allowing re-rewind.** The first instinct was "let rewind run multiple times — each rewind generates new mutations, so re-rewinding a rewound branch returns to the merged state and so on (per lesson #60)." But it gives the user a confusing UX ("did I rewind, or rewind back?") and burns mutation log space. The status guard refuses second rewind with a clear error; if the user wants to redo, they create a new branch with the same intent. One operation, one terminal state.

## M5.P4 point-of-no-return
134. **Plumb the safety flag end-to-end before any capability uses it, not after.** No existing M0–M5 capability is genuinely point-of-no-return — every current L2+ cap is bounded-reversible (graph mutations, branch ops, app promotion). It would have been easy to defer pointOfNoReturn until the first PONR cap landed (M5.P4.b: a future "git push" or "send email"). Instead the whole gate ships now: cap declaration → interceptor surface → Spotlight typed-confirm UI → tests. When the first real PONR cap lands, it adds ONE line (`pointOfNoReturn: true`) and the entire safety chain works. Lesson: when a contract crosses 3+ layers (cap → interceptor → UI → user), build the contract before the first user. Otherwise the first user's PR has to touch all four layers and review takes a week.
135. **Typed confirmation = "type the cap id" is better than "type DELETE" or a checkbox.** Three options were considered: (a) checkbox + Enter, (b) type the literal string "DELETE" / "CONFIRM", (c) type the cap.id (e.g. "git.push"). Option (c) wins because the typed text DOCUMENTS what the user is about to do — they can't muscle-memory through it without reading. The cap id is also already on screen above the input, so the cognitive load is "copy the displayed id." Friction is the point: PONR is rare enough that 5 seconds of typing is the right tax. Same pattern as GitHub's "type the repo name to delete it" gate.
136. **Mismatch-on-typed-confirm should bounce back, not abort.** First instinct: typing the wrong text aborts the interception. Worse UX — the user pressed Enter on a typo and now has to re-trigger the whole intent. Right behavior: clear the input, show "did not match" hint, keep the panel open. The user gets a second try without losing context. Aborting on mismatch is correct only if the typed text contains a hostile keyword pattern (and even then, log + abort + flag, don't silently re-prompt). For honest typos, bounce.

## M6.P1 red-team agent
137. **`IDBKeyRange.lowerBound(timestamp, true)` is EXCLUSIVE — `committedAt - 1` misses same-millisecond mutations.** branch-manager's `rewindBranch` queried `getMutationsSince(committedAt - 1)`. graph-store's `getMutationsSince` uses `IDBKeyRange.lowerBound(ts, true)` which means "strictly greater than ts." On rapid merge→rewind sequences (tests, especially), the merge mutations land in the same ms as `committedAt`, so `(committedAt - 1) + EXCLUSIVE = committedAt` excludes them. The bug only surfaced when the v03-verification suite was re-run frequently — single-run tests had enough wall-clock between phases to hide it. Fix: use `createdAt - 1` as the lower bound (always strictly before any mutation this branch produced) and rely on the capabilityId filter to scope. Generalisation: any time you query with a timestamp boundary, double-check whether the index is INCLUSIVE or EXCLUSIVE — they're not interchangeable, and rapid-fire tests are where the difference shows up.
138. **The verification suite's `astrion-budget-day` localStorage entry persists across page reloads and silently exhausts the executor's 50-token daily cap.** After ~50 test runs in a day, every L1+ step in section 6 starts failing with "plan exceeds daily budget (2 > 0 tokens)." This is correct production behavior (the cap is real safety), but wrong test behavior (the suite wants deterministic re-runs). Fix: `localStorage.removeItem('astrion-budget-day')` at the top of v03-verification.html alongside the existing `nova-ai-provider = mock` reset. Generalisation: every persistent-state test harness needs a "reset all my counters/caps/throttles" preamble, not just for what the test creates but for what surrounding kernel state it borrows.
139. **The red-team's only structural decision is: stub the LLM call so the test suite can exercise the full event-flow without an API key.** reviewAction's stub returns canned risks for `files.delete` (high severity, recommendation 'review') and empty risks for everything else. The verification suite tests the EVENT WIRING (interception:preview → red-team subscribes → emits interception:enriched), not the LLM's actual quality. Quality comes from soak-testing with a funded ANTHROPIC_API_KEY. Same pattern as the M3/M4 stubs — separate "wiring works" from "model produces useful output." Both are necessary; they're different test modes.
140. **The red-team subscriber must filter by cap.level >= 2 OR every L0/L1 op fires a wasteful AI call.** Without the level filter, every interception:preview would trigger a review — but L0/L1 caps DON'T trigger interceptions (interceptedExecute passes them through). The filter is defensive; the interceptor's level gate is the primary one. Keep both: the second check protects against future changes that might widen what counts as "interception-worthy." Lesson #117 generalised: defense at three boundaries beats one expensive check.

## M6.P4 rubber-stamp tracker
141. **A safety gate without a "did the user actually engage" telemetry is theatre.** M5/M6 builds layered safety (interceptor preview → red-team review → typed confirm for PONR), but if the user has muscle-memory'd Enter, the gates collapse to friction the user routes around. The rubber-stamp tracker is the missing feedback loop: classify each confirm as rapid (< 1.5s) or considered (>= 1.5s) and emit a Socratic warning when rapid-rate > 80% over 20+ samples. The warning is the only mechanism that lets the SYSTEM notice when its own affordances are being bypassed. Rule: every approval gate needs a "was this approval thought-through" probe + a feedback path back to the user when the answer is "no."
142. **A 24h cooldown on warnings is the difference between guidance and harassment.** Without cooldown, every L2+ confirm after the threshold would re-fire the warning. With cooldown, the user gets the message once a day max — annoying enough to register, not so annoying they disable the whole system. Same pattern as macOS battery low warnings, browser update prompts, etc. Bound the warning rate to the user's tolerance for being interrupted.
143. **Distinguishing user-abort from timeout-abort matters for trust scoring.** A user who manually aborts read the preview and decided no — that's GOOD engagement (high trust). A timeout-abort means the user walked away and never looked. They look identical at the event-bus level (both are interception:abort), but mean opposite things for the rubber-stamp metric. The tracker reads the `reason` string and counts them separately. Generalisation: any time you classify user behaviour from event-bus signals, look at the metadata, not just the event name. The same event with different reasons can mean opposite things.

Edit's old_string needs either the rendered emoji OR the literal 6-char escape; the tool tries swapping but fails on long blocks where other characters also differ subtly. Workaround: Python `open().read()` + string `.replace()` matches by raw bytes and never confuses escape sequences with characters. Rule: when Edit fails on a block containing escape sequences twice in a row, drop to Python sed-style replace rather than narrowing the match further. The bytes in the file are the source of truth.

169. **Cancelled demo ≠ failed demo. Different signal, different lesson.** The 2026-04-26 Sunday demo was cancelled, not run-and-broke. That means: the safety triple, the M8.P5 self-upgrade, the chat panel's 4 modes, the gpt-oss:16b path — all still UNVERIFIED end-to-end against a live audience. The 6 days of post-demo silence weren't recovery; they were "demo didn't happen, what now." Plan-of-record needs reconcile: (a) the demo prep work (lockfile fix, landing page, competitive brief, Phase 2 pull-forward) was correct work even though the trigger event evaporated, (b) the next external-facing milestone is now the launch of v1.0 in Dec, not a tomorrow demo, so the urgency-shape for fixes is "credibility for strangers" not "won't break in front of friends," (c) the real-AI soak gating was demo-driven; it's still gating but the deadline pressure shifted. 170. **Help indexes need categories once they pass ~10 entries.** Spotlight's `?` handler shipped 16 flat command rows. 171. **When the substrate is already there, the "feature" is three localStorage keys.** Sprint A (self-hosted AI brain picker) sounded like a multi-day build: setup wizard step + RAM detection + ndjson pull progress + provider wiring + slim-ISO path. Reality: `/api/ai/ollama-pull` already streams ndjson, `/api/system/memory` already returns MB-totals, `ai-service.js` already reads `nova-ai-provider` + `nova-ai-ollama-url` + `nova-ai-ollama-model` from localStorage, and the slim-ISO Ollama gap was a 4-line `build.sh` flip. The amount of code you actually need to write is the spec MINUS what's already shipped — and on a mature codebase that delta is usually 30%, not 100%. 172. **A full `render()` on every keystroke loses input focus — split the callback into "structural change" vs "lightweight sync."** The wizard's brain picker has a Remote URL input. Updating `state.remoteUrl` in the input handler had to enable the Continue button — but the button's `disabled` attribute is set at render time. Calling `render()` from the input listener wiped the focused input and the user couldn't type more than one character. Fix: have `onChange` accept a reason (`"input"` vs default), and the wizard's onChange branches: for `"input"` it imperatively updates only `#setup-next`'s `disabled`/`opacity`; for everything else it does a full re-render. Rule: any controlled input that gates other UI state should pair the state-write with an imperative DOM patch — re-rendering the whole tree on each keystroke is a UX bug masquerading as architectural purity.

173. **Idempotent service-start endpoints should probe-then-act, not act-then-probe.** `/api/ai/ollama-start` (added Sprint A) needs to be safe to call from the wizard regardless of build profile or host OS. First version called `sudo -n systemctl enable --now ollama` unconditionally — fine on slim Linux ISOs, but on macOS dev where there's no systemctl it would fail every call. Better: `fetch http://localhost:11434/api/tags` with a 1.5s timeout first; if it returns OK, return `{ok:true, alreadyRunning:true}` immediately. Only fall through to systemctl when the daemon isn't responding. Saves the cost of systemctl on the happy path AND makes the endpoint trivially safe on hosts without systemd. Rule: any "start X" endpoint should check whether X is already running before trying to start it — both for performance (1.5s probe vs 200ms+ systemctl spawn) and for portability (the probe works everywhere; the imperative often doesn't).

174. **Soft-warn for unknown sizes; only hard-block fits-the-numbers misses.** Sprint B's RAM gate had three real states: known-size + known-RAM + fits → silent pass; known-size + known-RAM + doesn't fit → modal with explicit "Pull anyway"; everything else (size unknown OR RAM unknown) → modal saying "we don't know — pulling will succeed but running might swap." The naïve version was binary "fit or block," which would have hard-blocked any user who typed a model name not in our lookup table. Wrong: an unfamiliar model name is data-uncertainty, not danger — you can't refuse on grounds you don't have. Rule: when a safety gate's input is fuzzy (lookup miss, OS without `/proc/meminfo`, etc.), the response is a soft warning with a forward path, not a hard block. Hard blocks belong to "we know the bad outcome and chose against it"; soft warnings belong to "we can't see the outcome — here's what to watch for."

175. **Diagnostics endpoints should fail soft per-field, not abort on any subsystem.** `/api/ai/ollama-status` had to gather four independent things: liveness probe (`/api/tags`), `systemctl is-active`, `journalctl -u ollama`, `df`. First-pass instinct was a single try/catch wrapping all four — but each failure mode is independent: macOS dev has no systemctl, a slim ISO without ollama installed has no journal, a freshly-booted box might have an empty journal but a running daemon. Final version: separate try/catch (or graceful runShell call) per subsystem, each populating one field of an output object initialized with sensible nulls. The Diagnostics panel then paints whatever it got — "RAM unknown · 966 GB free disk · 2 models pulled" still beats a 502. Rule: when an endpoint composes results from N independent best-effort sources, the failure boundary is per-source, not per-endpoint. The total response's job is to tell the truth about what's known, including what isn't.

176. **The "edit four places to add one app" smell stays a smell until you actually extract.** boot.js had three identical 76-line register-call blocks (Spotlight popup mode, native-app mode, normal desktop boot) — sanity-check #2 flagged it as a "maintenance hazard," and that was three weeks before this session. The fix was a 35-line `registerAllApps()` helper plus three call-site replacements: 76 lines × 3 blocks → 1 list. Net diff: −146 lines, +40 lines, single source of truth. Risk: refactor must reproduce the SAME side effects, in the SAME order — 76/76/76 calls in 76/76/76 identical orderings, verified by a 5-line Python diff before edit. Why it sat: the smell looks expensive ("touch boot.js"), but the refactor was 15 minutes of careful editing once the diff confirmed all three blocks were identical. Rule: when you flag a "drift hazard" in an audit, the next session should EITHER fix it OR remove it from the hazard list. Letting hazards accumulate signals "I'd rather complain than commit" — which is its own bad signal to the next reader of the audit.

177. **`err.message` going into `innerHTML` is XSS, even when the source feels "internal."** Sanity-check flagged appstore.js for this; a quick grep across `js/apps/` showed the same pattern in activity-monitor (3 sites) and settings (2 more) — six places total where a thrown error's `.message` was string-interpolated into a template literal that ended up in `innerHTML`. The "but it's just an internal error" defense fails in two cases: (1) errors from `fetch()` against an upstream service surface the upstream's response text, and Astrion talks to Ollama (third-party) + Flathub (third-party) + Waydroid + the Anthropic API — all attacker-influenceable in the right pwn chain. (2) `JSON.parse(...)` errors include the offending source text in their message. So `err.message` is in the same trust class as untrusted-user-input. Fix: `js/lib/safe-html.js` exports `escapeHtml(s)` and `escapeError(err)`; threaded through all six sites. Existing inline pattern (`textContent → .innerHTML` ping-pong) was correct but verbose; the helper makes the intent explicit and shorter. Rule: any time you grep `${err.message}` or `${e.message}` inside an `innerHTML` template, count it as XSS until proven otherwise. The "is this attacker-controlled?" question is much harder to answer correctly than the "let me just escape it" reflex.

178. **Measure before you optimize — instrument the boot now so the perf target has a number to chase.** ROADMAP-DEC-2026-v3.md and sanity-check both list "boot time <1.5s" as a Phase 1 goal, and both describe it as "unmeasured." Adding a 20-line timing harness (`__bootMark`/`__bootFlush` + a `localStorage` persist + a System Info row) took 15 minutes and revealed the actual numbers on macOS dev: boot:start → kernel:ready = 3.2 s of which ~1 s is filesystem/graph init, ~1 s is the 76-app register sweep, ~1 s is window-manager wiring. Login wait dominates wall-clock time (`login:complete` at 10s) but isn't real boot work. Without the instrumentation we'd be optimizing blind — "shave 200 ms off boot" is meaningless without per-phase data. Rule: when a perf target sits in a roadmap as "unmeasured," the FIRST commit toward it is the measurement, not the optimization. The data tells you which phase to attack; intuition about "what's slow" is wrong half the time.

180. **A red-team partner that ALWAYS recommends "review" or "abort" gates the entire happy-path demo to manual override — RESOLVED with a structural fix.** The M8.P5 5-gate machinery used to treat anything other than `recommendation: "proceed"` as a gate failure. 181. **The pen-test caught a real bug the unit tests missed: applyProposal's defense-in-depth content scan was load-bearing in the high-level wrapper only.** Built `js/kernel/pen-test.js` (synthetic red/blue/purple team suite) on 2026-05-03. First run: 5/6 green. Test #3 (content-blocklist) FAILED — 4 attack patterns (`eval(`, `new Function(`, `localStorage.removeItem('astrion-…')`, `graphStore.deleteNode(`) all slipped through the apply pipeline when the proposal was staged via the lower-level `proposeSelfMod()` and applied via `applyProposal()`. The high-level `proposeUpgrade()` calls `checkContentSafe()` BEFORE staging, so production was safe — but a caller that bypassed proposeUpgrade got past every gate. The pen-test made the gap visible. 182. **An AI without persistent memory is a goldfish — and the fix is one localStorage key.** The chat panel and Spotlight kept conversation context in `aiService.conversationHistory` (in-memory only). 183. **The "76-app register sweep" was a red herring; cosmetic boot animations were the bottleneck.** Lesson #178 instrumented boot.js and read 3.2 s to kernel:ready on macbook dev — the handoff blamed the 76-app register sweep as "the primary attackable phase." Re-measuring on browser preview: boot:start → kernel:ready was ~905 ms, of which **900 ms was four `await animate(progressBar, %, 400/300/200/300)` calls plus a `sleep(400)` before the boot-screen fade**. `processManager.register()` is one `Map.set` — 76 of those run in microseconds. `windowManager.init()` is two event listeners. The "phases each take ~1 s" reading was almost entirely the artificial CSS-transition awaits between marks. Fix: replaced `animate()` (resolved-after-N-ms timer) with `setProgress()` (sets bar width and lets the CSS transition run in parallel with real work). Cut `sleep(400)` before fade to `sleep(120)`; cut fade transition from 500 ms to 250 ms; killed an unused `sleep(300)` before `desktop:ready`. Net: kernel:ready at **7 ms** in browser preview cold boot (post-IndexedDB-wipe), down from 905 ms. v03 216/216 still green, golden 18/18, console clean. Rule: a perf instrumentation that makes a phase look "1 s wide" needs a control test before you optimize the work — was the second longest? maybe the wait between two marks IS the wait, not the work. The simplest first probe: temporarily remove every `await sleep`/`await animate` between marks and re-measure. If the phase collapses, the wait was the cost. Corollary: when the fix is "stop blocking on a CSS transition that runs in parallel anyway," that's not a hack — `await new Promise(r => setTimeout(r, 400))` after `bar.style.width = '30%'` is a literal no-op for the user; the bar already started transitioning the moment you set the style.
